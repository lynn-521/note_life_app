# 综合家庭管家产品设计文档

> 版本：v0.1（设计蓝图）
> 主形态：Flutter 移动 App ｜ 微信提醒：多渠道可插拔 ｜ 存储：本地优先 + 云端同步
> 适用人群：家庭单位（多成员共享）

---

## 1. 产品定位与核心价值主张

### 1.1 一句话定位
**一个把「家里的东西、该吃的药、今晚的菜、要去的行程」全部管起来，并通过微信主动提醒你的家庭管家。**

### 1.2 要解决的真实痛点
| 痛点 | 现状 | 本产品的解法 |
|---|---|---|
| 东西买了忘 | 冰箱/药箱塞满，临期才发现 | 入库即记录批次与有效期，临期自动微信提醒 |
| 重复购买 | 家人各买各的，囤货重叠 | 共享库存，低库存才提醒补货，避免囤多 |
| 吃药靠记 | 老人/孩子用药全凭记忆，容易漏服 | 按成员排用药计划，到点微信提醒 + 服用记录 |
| 今晚吃啥 | 每天临时想，重复率高 | 家庭菜谱库 + 每日菜谱，标记「谁会做」 |
| 出游手忙脚乱 | 行程/行李/预算散落各处 | 结构化旅游计划书，统一查看与勾选 |

### 1.3 核心价值主张
- **主动提醒，而非被动查询**：把「人找信息」变成「信息找人」，关键节点推到微信。
- **一家人共用一个家**：成员、库存、菜谱、行程在家庭空间内共享。
- **离线也能用，联网自动合**：本地优先，地铁里也能记一笔，回家联网自动同步。

## 2. 目标用户与典型场景

### 2.1 目标用户
- **核心用户**：25–55 岁家庭主力（如峰哥这类「家庭 IT 管理员」），负责采购、备药、安排饮食。
- **协同用户**：配偶、父母、子女——共享库存、认领菜谱、接收提醒。
- **重点照顾对象**：需要长期服药的老人、需补充营养品的家人。

### 2.2 典型场景
1. **临期拦截**：妈妈买牛奶入库（批次 + 保质期 7 天）。第 5 天，微信收到「牛奶还有 2 天到期，尽快食用」。
2. **药品预警**：家中常备药「布洛芬」仅剩 2 盒，低于阈值，微信提醒「布洛芬库存偏低，建议补货」。
3. **用药打卡**：父亲每天早 8 点、晚 8 点服降压药。到点微信提醒；子女看到后确认父亲已服，记录留痕。
4. **今晚菜单**：周末晚上，App 根据菜谱库和现有库存推荐 3 道菜，标记「 wife 会做红烧肉、老公会做番茄炒蛋」，一键排成明日菜单。
5. **出游筹备**：五一出行前建旅游计划书，列出 3 天行程、行李清单（充电宝/证件）、预算 3000，家人各自勾选负责项。

## 3. 功能架构（模块划分）

```
┌──────────────────────────────────────────────────────────┐
│                      综合家庭管家 (Flutter App)              │
├──────────┬──────────┬──────────┬──────────┬───────────────┤
│ A 进销存  │ B 提醒中心│ C 用药营养│ D 菜谱管理│ E 备忘录      │
│  (核心)   │ (规则引擎)│ (按成员) │          │ F 旅游计划书   │
├──────────┴──────────┴──────────┴──────────┴───────────────┤
│ G 成员与权限（共享空间底座：管理员 / 成员）                  │
├──────────────────────────────────────────────────────────┤
│ 微信提醒联动层：NotificationChannel 可插拔（见第 8 章）      │
└──────────────────────────────────────────────────────────┘
```

### A. 家庭仓库进销存（核心）
- 商品管理：名称、分类（食物 / 药品 / 日用品）、单位、条码、存放位置、低库存阈值。
- 入库：扫码或手动，生成入库单 + 库存批次（含有效期）。
- 出库：消耗 / 丢弃 / 其他，生成出库单；库存 = 入库流水 − 出库流水（事件聚合，见 6.4）。
- 双预警：**临期预警**（有效期前 N 天）、**低库存预警**（低于阈值）。
- 重点品类：食物、药品置顶关注，提供「即将过期」「需补货」快捷视图。

### B. 微信提醒中心
- 规则引擎：把 A–F 各类「需要被提醒的事」抽象成 `ReminderRule`。
- 触发源：临期、低库存、用药时间点、每日菜谱推送、自定义。
- 下发：经 `NotificationChannel` 路由到微信（或个人/群组）。
- 日志：`ReminderLog` 记录每次下发结果，便于排查漏提醒。

### C. 用药与营养品管理（按成员区分）
- 每个成员独立的用药计划：药名 / 类型（药 / 营养品）/ 剂量 / 频次 / 时间点。
- 到点提醒：微信推送「XX 该服用 XX（剂量）」。
- 服用记录：已服 / 跳过 / 备注，形成可追溯的健康打卡。
- 家庭视角：子女可查看父母用药记录，发现连续漏服可介入。

### D. 菜谱管理
- 家庭菜谱库：菜名、步骤、食材清单、份量、标签、创建人。
- 每日菜谱：按日期 + 餐别（早 / 午 / 晚 / 加餐）排菜。
- **谁会做（多人标签）**：`RecipeCookableBy` 多对多，一道菜可标记多个家庭成员会做。
- 食材反查：菜谱食材关联商品，排菜单时提示「缺少的食材」，顺手生成补货项。

### E. 家庭备忘录
- 轻量便签 / 待办，支持家庭成员。
- 置顶、完成态、到期提醒（可挂接提醒中心）。

### F. 旅游计划书
- 结构化模板：行程天数、每日安排、行李清单、预算、参与成员。
- 成员可认领行李项 / 预算项，出行前统一勾选核对。

### G. 成员与权限
- 家庭成员账户，角色：管理员 / 普通成员。
- 共享家庭空间：库存、菜谱、行程在家庭内可见可协作。
- 微信登录（fluwx）+ 云端空间绑定。

## 4. 核心业务流程

### 4.1 入库 → 库存聚合 → 预警
```
扫码/手动录入商品
   └─> 生成 InboundOrder（入库单）
         └─> 生成 StockBatch（批次 + 有效期 + 数量）
               └─> 库存视图 = Σ入库 − Σ出库
                     └─> 定时扫描：
                          · expireDate − today ≤ 阈值 → 临期规则触发
                          · 当前库存 ≤ 低库存阈值 → 低库存规则触发
                                └─> ReminderRule → NotificationChannel → 微信
```

### 4.2 用药提醒与记录
```
Member 创建 Medication（用药计划）
   └─> 生成 DoseSchedule（每日时间点）
         └─> 到点（本地定时器 / 云端 cron 兜底）
               └─> 微信提醒「该服药」
                     └─> 家庭成员确认 → DoseLog(status=done, takenAt)
                          └─> 连续漏服 → 家庭视角预警
```

### 4.3 每日菜谱生成
```
家庭成员维护 Recipe 库（含谁会做）
   └─> 排 DailyMeal（日期 + 餐别 + Recipe）
         └─> 反查食材库存，提示缺口
               └─> 每天固定时间微信推送「今日菜单」
```

### 4.4 提醒状态机（通用）
```
ReminderRule(enabled)
   └─> 触发条件命中
         └─> 生成待发任务
               ├─ 本地在线 → 直接经 Channel 下发
               └─ 本地离线 → 云端 cron 兜底下发
                     └─> ReminderLog(status: sent/failed)
                           └─ 失败重试 / 告警
```

### 6.1 实体关系（ER，概念图）
```
Member ──< Medication >─── DoseSchedule ──< DoseLog
```

### 6.3 通用字段（所有实体）
`id`(PK)、`createdAt`、`updatedAt`、`version`(版本向量/递增)、`deletedAt`(软删，便于同步)。

### 6.4 库存聚合与冲突合并（关键设计）
- **不存库存绝对值，只存流水**：库存 = Σ InboundOrder.qty − Σ OutboundOrder.qty。多端并发只新增事件，天然无冲突。
- **合并策略**：以 `updatedAt` + `version` 做字段级 last-write-wins；流水类实体为 append-only，合并即并集。
- **删除**：统一软删（`deletedAt`），同步时按时间戳处理。

### 7.2 模块职责
- **UI 层**：各功能模块页面，纯展示与交互。
- **Riverpod**：状态与业务逻辑，UI 不直接碰 DB。
- **本地 DB（Isar/Drift）**：离线优先数据源，所有读写先落本地。
- **同步引擎**：监听本地变更 → 推云端；拉取远端变更 → 合并（6.4 策略）。弱网/离线时进队列，联网重放。
- **本地提醒调度器**：负责「App 在线时」的到点提醒（用药、每日菜谱、临期扫描）。
- **云端云函数**：定时 cron 扫描全量规则，对「App 不在线/未确认」的提醒兜底经微信下发。
- **NotificationChannel**：提醒统一出口，按规则选渠道下发（第 8 章）。

## 8. 微信提醒联动方案

### 8.1 设计原则：可插拔
定义统一接口，新增渠道只需实现 `send()`，业务层无感知：

```
abstract class NotificationChannel {
  ChannelType type;
  Future<SendResult> send({
    required List<String> targets,   // 接收人 wxUid / 群 webhook
    required String title,
    required String body,
    Map<String, dynamic>? payload,
  });
}
```

### 8.2 渠道矩阵（推荐层级）
| 渠道 | 实现类 | 门槛 | 成本 | 适用 | 推荐度 |
|---|---|---|---|---|---|
| **WxPusher**（个人微信） | `WxPusherChannel` | 低（建 App 拿 token） | 免费 | 推到个人微信，零门槛首选 | ★★★★★ 默认 |
| Server酱 / PushPlus | `ThirdPartyChannel` | 低 | 免费/限流 | 个人微信推送备选 | ★★★★ |
| 企业微信群机器人 | `GroupBotChannel` | 低（建群+Webhook） | 免费 | 推到家庭微信群 | ★★★★ |
| 公众号模板消息 | `OAChannel` | 高（认证服务号） | 付费认证 | 正式品牌推送 | ★★ |
| 小程序订阅消息 | `MiniProgramChannel` | 中（需小程序） | 免费 | 原生订阅推送 | ★★★ |

> 结论：**MVP 默认 WxPusher + 企业微信群机器人**（个人/家庭零成本）；公众号/小程序作为进阶按需接入。

### 8.3 各渠道接入要点
- **WxPusher**：官网建应用 → 家庭成员扫码关注并绑定 UID → 调用 REST 发送（按 topic/UID）。支持图文、点击回链。
- **企业微信群机器人**：群设置添加机器人 → 取得 Webhook URL → POST markdown 消息即可，适合「家庭群广播」。
- **公众号模板消息**：需认证服务号 + 模板库；用户 48h 内互动或走订阅通知。
- **小程序订阅消息**：需已上线小程序；用户一次性/长期订阅后，服务端调接口下发。

### 8.4 触发与兜底
- **在线即推**：本地调度器命中规则 → 直接走 Channel。
- **离线兜底**：云端 cron 周期性扫描未确认/未下发的提醒 → 经 Channel 下发，保证「人不在 App 旁也能收到微信」。
- **去重与限频**：同一规则 N 分钟内不重复推；`ReminderLog` 记录状态，失败自动重试。

## 9. 技术选型

### 9.1 推荐栈（贴合峰哥的 Flutter 经验）
| 层 | 选型 | 理由 |
|---|---|---|
| 语言/框架 | **Flutter (Dart)** | 与 LureLife 一致，跨端一套代码 |
| 状态管理 | **Riverpod** | 编译安全、可测试、适合中大型 |
| 本地 DB | **Isar** 或 **Drift** | 离线优先、类型安全；Drift 偏 SQL、Isar 偏 NoSQL，按团队偏好 |
| 云后端 | **CloudBase（腾讯云）** 或 **Supabase** | 实时同步 + 云函数；CloudBase 天然微信登录更顺 |
| 微信登录 | **fluwx** | LureLife 已验证，直接复用 |
| 本地调度 | **flutter_local_notifications + WorkManager** | 离线到点提醒 |
| 云端定时 | **云函数 cron** | 兜底微信推送 |
| 图表/报表 | **fl_chart** | 库存/用药趋势可视化（可选） |

### 9.2 关键取舍
- **CloudBase vs Supabase**：优先 CloudBase（微信生态闭环、登录简单）；若偏好开源/SQL 选 Supabase。
- **Isar vs Drift**：偏好 SQL 与复杂查询选 Drift；偏好性能与简洁 API 选 Isar。
- **自研同步 vs 平台实时**：CloudBase/Supabase 自带实时通道，优先复用，减少自研成本。

## 11. 关键决策、风险与权衡

### 11.1 关键决策
1. **库存用流水聚合而非存绝对值** —— 多端并发零冲突，合并简单。
2. **本地优先 + 云端兜底提醒** —— 离线可用，且 App 不在线也能收到微信。
3. **用药按成员建模** —— 每人独立计划与记录，避免家庭混用、方便照护。
4. **菜谱多厨师标签（多对多）** —— 支持多人会做同一道菜，排菜更灵活。
5. **提醒渠道可插拔** —— 不被单一微信渠道绑定，按需切换/扩展。

### 11.2 风险与缓解
| 风险 | 影响 | 缓解 |
|---|---|---|
| 微信第三方推送限流/停服 | 提醒失效 | 多渠道冗余（WxPusher + 群机器人）；核心提醒本地通知兜底 |
| 多端同步冲突 | 数据错乱 | 流水聚合 + 字段级 LWW + 软删 |
| 云端成本 | 费用 | MVP 用免费额度；CloudBase/Supabase 入门档足够家庭量级 |
| 隐私（家庭健康/用药） | 泄露 | 家庭空间内最小权限；敏感数据本地可仅存本地 |
| 用药提醒合规 | 误导医疗 | 产品定位「提醒/记录工具」，不提供医疗建议 |

### 11.3 待确认项（默认假设）
- 用药与营养品**按成员独立**建模（已采纳）。
- 旅游计划书采用**结构化模板**（行程/行李/预算），保留自由备注（已采纳）。
- MVP 优先级：进销存 + 临期/药品提醒 + 用药记录 + 微信推送先行；菜谱/备忘录/旅游为 P1/P2（已采纳）。
